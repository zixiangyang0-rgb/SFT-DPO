# 主流模型微调方案面试话术（1000字速记版）

> 适用：NLP/大模型算法岗、AI 应用岗；核心套路：**讲人话 + 讲对比 + 讲选择理由**。

---

## 一、30 秒开场（先说全貌）

> "我对主流微调方案的理解可以浓缩成一句话：**Pre-train 决定智商，SFT 决定'懂规矩'，DPO/RLHF 决定'做好人'**。SFT 教的是'说什么'，DPO 教的是'哪种说法更好'。我平时做项目主用 SFT + DPO 两阶段，简单任务直接 SFT 一次，复杂任务上 ORPO（把 SFT 和 DPO 合并成一步，省显存还不用 Reference Model）。**工具链上，原理研究用 Hugging Face TRL，个人单卡用 Unsloth（快 2-5 倍），业务方零代码用 LLaMA-Factory，多卡分布式才上 DeepSpeed。**"

---

## 二、SFT / DPO / PPO / ORPO 怎么选？（方案选型题）

面试官最常问的对比题，**直接套表回答**：

| 方案 | 本质 | 显存 | 难度 | 适用 |
|---|---|---|---|---|
| **SFT** | 喂 (Instruction, Output) 标准答案 | 低 | 最低 | 教模型"按规矩说话" |
| **DPO** | 喂 (Chosen, Rejected) 让模型挑好的 | 中（要 Reference Model） | 中 | 教模型"什么是更好的回答" |
| **PPO** | 强化学习，要训 Reward Model + Critic | 极高 | 极难 | 大厂旗舰产品，**个人开发者基本不用** |
| **ORPO** | 把 SFT 和 DPO 写进同一个 Loss | **低（不要 Reference Model）** | 中 | 单卡显存吃紧 + 想少训一轮 |

**金句**："**SFT 是铺地基，DPO 是装修，PPO 是把房子推倒重盖**。地基本身没盖好，装修再漂亮也是空中楼阁。"

---

## 三、SFT vs DPO 的本质区别（灵魂拷问题）

> **一句话**："SFT 教'能说话'，DPO 教'说好话'。"
>
> **故事化表达**：SFT 像师父在旁边手把手教菜谱——客人说'要辣的'你就给麻婆豆腐。徒弟学会了**格式**。但他有时会往菜里加'砒霜'（有毒），或虽然熟了但又咸又难吃。DPO 就像你端出 A、B 两盘菜说'选 A 别选 B'——只教偏好，不教答案。
>
> **数据差异**：SFT 是 (Instruction, Output) 一对一；DPO 是 (Prompt, Chosen, Rejected) 三元组，**必须同时有正例和负例**，模型通过对比才能学出"价值观"。

---

## 四、SFT 数据怎么准备？（高频踩坑题）

**70% 取决于数据，30% 取决于算法。** 讲 3 个核心点：

1. **格式选型**：Alpaca（单轮一问一答）vs ShareGPT（多轮对话）。**直接 All in ShareGPT**——单轮也能包装成只有一轮的 conversations，脚本只写一套。

2. **清洗三板斧**：
   - **去重**：精确去重 + MinHash+LSH 模糊去重（"今天天气不错" vs "今天天气真不错" 算相似）；
   - **去"AI 味"**：GPT-4 生成的数据里一堆 "As an AI language model..."，**整条删掉**，否则模型会变成道歉复读机；
   - **关键词过滤**：暴力色情广告。

3. **黄金配比 7-2-1**：70-80% 垂直领域 + 10-20% 通用（防灾难性遗忘）+ 5-10% 自我认知（"我是XX助手"，防止它说"我是 ChatGPT"）。

> **加分话术**："**LIMA 论文证明 1000 条精修数据吊打 5 万条粗制数据**。我宁可花一周精标 500 条，不愿爬 5 万条脏数据。"

---

## 五、为什么用 LoRA / QLoRA？（技术深度题）

**一句话**："全量微调要更新所有参数，**显存是模型大小的 4 倍以上**，7B 模型单卡根本跑不动；还会'灾难性遗忘'，把旧知识冲掉。"

**LoRA 哲学**：冻结原模型，**只外挂两个小矩阵 A 和 B** 学习新知识。**形象比喻**——模型是一本百科全书，全量微调是把字擦掉重写（费劲、容易写坏）；LoRA 是在书页边角**贴便利贴**（轻松、随时可撕）。数学上 ΔW = A × B，A 和 B 参数量只有原模型的 0.1%-1%。

**QLoRA = 4-bit 量化基座 + LoRA 适配器**。7B 模型原本 14G 显存，4-bit 量化后只占 3.5G，**单卡训练标准答案**。

---

## 六、核心参数口诀（务必背下来）

- **Rank**：简单任务 8，复杂任务 64，**16 是性价比之选**；
- **Alpha = 2 × Rank**（无脑套公式）；
- **Learning Rate**：`2e-4`（LoRA 比全量微调高一个数量级，因为可训参数少）；
- **Target Modules**：**全部线性层**（q/k/v/o + gate/up/down），比只挂 q/v 效果提升明显，显存增加不多；
- **NEFTune**：加 `neftune_noise_alpha=5` 给 Embedding 加噪，**亲测显著提升指令遵循**；
- **Loss 曲线判断**：Loss 不是越低越好。**Loss=1.3 反而比 1.8 差**——它在背训练集。正确判断 = "Loss 下降 + 验证集不反弹 + 真实样本稳定"。

---

## 七、DPO 数据三大剧毒（避雷题）

DPO 对数据极其敏感，**踩一个模型就废**：

1. **长度偏见**：如果 Chosen 总比 Rejected 长，模型会变成话痨——**刻意保留一些"短而精"的 Chosen**；
2. **用 DPO 修事实错误** = 大错特错。**DPO 修的是风格和安全，不是知识**。知识要在 Pre-train 或 SFT 阶段注入；
3. **相似度过高**：Chosen 和 Rejected 只差一两个字 → Loss 接近 0 → **梯度消失学不到东西**，必须保证明显的逻辑/语气差异。

**数据来源**：人工标注（OpenAI/Anthropic 做法，最贵）；**LLM-as-a-Judge**（用 GPT-4 当裁判，**工业界主流**）。

---

## 八、ORPO 为什么值得了解？（加分题）

> "传统流程是 Base → SFT → DPO 两阶段训练，**训两遍、加载 Reference Model、显存翻倍**。ORPO 把 SFT Loss 和 OR Loss 写进同一个公式，**一次训练搞定，且不需要 Reference Model**——7B 模型 DPO 要 30G 显存，ORPO 只要 20G。**单卡玩 DPO 的人，ORPO 是首选。**"

### SFT 数据准备篇

| 追问 | 一句话回应 | 说人话展开 |
|---|---|---|
| **SFT 数据多少条够？** | 1000-10000 条精标，**LIMA 论文已证明质量 > 数量** | "很多人觉得数据越多越好，其实这是个误区。LIMA 论文做过一个实验：只用 1000 条顶级人类精修数据微调出来的模型，直接吊打用 5 万条粗制数据训的模型。我的经验是，垂直领域 1000-10000 条精标数据完全够用，关键是数据要干净——格式统一、没有错别字、标签准确。我宁可花一周精标 500 条，也不愿爬 5 万条脏数据喂给模型。" |
| **数据从哪里来？怎么构造？** | 垂直数据自建 + 通用数据用开源 + 自我认知自己写 | "实话说，数据来源分三块：垂直领域的数据（比如医疗、法律）我主要靠业务侧提供的真实问答，或者请标注员按照我们的风格指南来标；通用能力这块我直接用 alpaca_gpt4_zh 和 ShareGPT_Vicuna，这些开源数据集已经清洗过了；至于'我是XX助手'这种自我认知数据，必须自己写脚本生成，大概几百条就行，告诉模型它是谁、它的能力边界、它的说话风格。" |
| **数据清洗怎么做？** | 去重 + 去 AI 味 + 关键词过滤，三板斧 | "清洗我就用三板斧。第一，精确+模糊去重，精确去重用字符串匹配删完全一样的，模糊去重用 MinHash+LSH，把'今天天气不错'和'今天天气真不错'这种高度相似的句子识别出来删掉其中一条。第二，去 AI 味——GPT-4 生成的数据里一堆'作为AI语言模型我无法回答'，这种整条删掉，否则模型会变成道歉复读机。第三，关键词过滤，过滤暴力色情广告代码。整个流程我一般用 Python 脚本跑，2000 条数据大概 10 分钟搞定。" |
| **格式用 Alpaca 还是 ShareGPT？** | **All in ShareGPT**，单轮也能包装成一轮对话 | "我 All in ShareGPT。以前我也分别维护两套格式，后来发现太麻烦了——单轮问答完全可以包装成只有一轮对话的 ShareGPT 格式，一个脚本处理所有数据，不用写两套清洗逻辑，框架兼容性还好。LLaMA-Factory、Unsloth、TRL 全都支持。" |

### 模型训练篇

| 追问 | 一句话回应 | 说人话展开 |
|---|---|---|
| **训练 Loss 不降怎么办？** | 90% 是数据问题——格式错乱、字段没对齐、混进脏数据 | "这个我踩过很多次坑，经验是 90% 的情况都是数据问题，不是模型问题。排查顺序是：第一步，检查 JSON 格式对不对——字段名是不是拼错了，是不是多了个空格；第二步，看 input 和 output 字段有没有错位；第三步，跑个统计分析，看有没有超长文本、异常字符、重复内容。如果数据查完了没问题，再去看学习率是不是太高了、模型基座是不是太小了。总之遇到 Loss 不降，先去清洗数据，别急着调参数。" |
| **怎么防止灾难性遗忘？** | 加 10-20% 通用数据 + 5-10% 自我认知数据 | "灾难性遗忘就是模型学完新知识后，把旧能力丢了——比如之前会写代码，微调后只会聊医疗了。解决方案是数据配比要讲究：70-80% 垂直领域数据用来学业务知识，同时掺 10-20% 通用能力数据（比如 alpaca_gpt4_zh 的通用问答），维持模型的原有智商；另外再加 5-10% 的自我认知数据，就是告诉它'你是XX医疗助手，不是ChatGPT'，防止它乱认身份。核心思想就是**别让模型偏科**。" |
| **LoRA 的 Rank 怎么选？** | 简单任务 8-16，复杂任务 64，**16 是性价比之选** | "Rank 决定了 LoRA 矩阵能学多少东西。简单任务比如改个说话风格、做分类，选 8 或 16 就够了；复杂任务比如学全新的专业知识、复杂推理，上 64 或 128。但 Rank 越大显存越高、越容易过拟合。我日常开发基本无脑用 16 或 64——16 用于快速验证想法，64 用于正式训练，64 是性价比最高的选择。配合 Alpha = 2×Rank，基本不用动脑子调。" |
| **NEFTune 是什么？有用吗？** | 给 Embedding 加噪声，**亲测显著提升指令遵循** | "NEFTune 全称 Noisy Embeddings Fine-Tuning，原理就是在训练时给输入的 Embedding 向量随机加一点点噪声。它的效果是防止模型死记硬背训练集，提升泛化能力。我自己用它调过一个对话模型，加上 neftune_noise_alpha=5 之后，指令遵循能力明显变好——之前模型偶尔会答非所问，用了之后回复更贴合用户意图。代码就一行，在 Trainer 里加个参数，几乎没有额外开销，推荐试试。" |
| **QLoRA 精度损失大吗？** | NF4 量化，**微调任务几乎可忽略**，推理可反量化回 FP16 | "说实话 QLoRA 的精度损失在微调任务里基本感知不到。它用的是 NF4 格式，4-bit 量化，7B 模型从 14G 显存直接压缩到 3.5G，单卡 3090 就能跑起来训练。量化只作用于基座模型，LoRA 适配器还是用 FP16 精度学习新知识，所以微调效果几乎不受影响。推理的时候，基座模型可以反量化回 FP16 或 BF16，也不会损失太多精度。**这是目前单卡训练的事实标准**。" |

### 模型评估篇

| 追问 | 一句话回应 | 说人话展开 |
|---|---|---|
| **怎么评估微调效果？** | LLM-as-a-Judge 主观打分 + 业务指标 A/B 测试双轨 | "评估分两层。**主观层**用 LLM-as-a-Judge，就是让 GPT-4 对微调前后的回答做对比打分，问它'哪个回答更好？为什么？'——这对评估对话风格、指令遵循特别有用。**客观层**必须看业务指标，比如在真实业务场景里做 A/B 测试，看转化率、用户满意度有没有提升。只看 Loss 曲线判断模型好不好是片面的——Loss 低不代表模型真的变强了，可能只是背住了训练集。" |
| **Loss 曲线怎么看？** | Loss 不是越低越好，**Loss 下降 + 验证集不反弹 + 真实样本测一遍** | "很多人以为 Loss 越低越好，其实不是。我的判断标准是三合一：训练 Loss 持续下降，说明模型在学习；验证集 Loss 不反弹、不飙升，说明没有严重过拟合；最重要的，拿真实业务样本丢进去跑一遍，看回答质量有没有提升。曾经我见过 Loss=1.3 的模型反而比 Loss=1.8 的差——它在背训练集。所以一定要结合真实样本测试，Loss 只是参考，不是标准。" |
| **微调后的模型和 GPT-4 比怎么样？** | 比不了也没必要比，**专精任务上小模型可以打赢大模型** | "我直接说实话：通用能力上，小模型肯定打不过 GPT-4。但微调的价值在于**专精**——在特定领域、特定任务上，7B 的微调模型完全可以打赢 GPT-4。比如你要做一个法律咨询助手，用几千条高质量法律问答微调出来的模型，在法律条款引用、案例分析这些任务上，比 GPT-4 的通用回答更准确、更符合业务需求。面试的时候要理直气壮地说：**我们不比通用，我们比专精**。" |

### 对齐与进阶篇

| 追问 | 一句话回应 | 说人话展开 |
|---|---|---|
| **DPO 和 SFT 到底怎么选？** | 简单任务 SFT 一次，复杂任务 SFT + DPO 两阶段 | "我的选择逻辑很简单：**任务简单就只做 SFT，任务复杂就上 SFT + DPO 两阶段**。SFT 的作用是教模型'说什么'——给指令，给标准答案，让它学会按规矩输出。DPO 的作用是教模型'哪种说法更好'——给正例和负例，让它学会偏好和边界。如果你只是要模型学会回答特定领域的问题，SFT 够了；如果你还需要模型说话有礼貌、不乱讲、不答非所问，上 DPO。ORPO 是我的备选——如果显存不够、又懒得训两遍，我会直接用 ORPO 一次搞定，效果也不错。" |
| **DPO 三大剧毒是什么？** | 长度偏见 + 事实错误乱用 + Chosen/Rejected 相似度高 | "DPO 对数据极其敏感，踩一个就废。**第一毒：长度偏见**——如果 Chosen 总比 Rejected 长，模型会变成话痨，为了讨好奖励机制疯狂堆砌废话。解药是刻意保留一些'短而精'的 Chosen，别让模型以为长就是好。**第二毒：用 DPO 修事实错误**——DPO 修的是风格和安全，不是知识。知识要在 Pre-train 或 SFT 阶段注入，指望 DPO 教模型正确知识是大错特错。**第三毒：Chosen 和 Rejected 相似度过高**——只差一两个字，Loss 接近 0，梯度消失，学不到东西。必须保证两个回答在逻辑、语气或结构上有明显差异。" |
| **为什么选 ORPO 而不是 DPO？** | 不要 Reference Model，省一半显存，**单卡首选** | "DPO 需要同时加载原模型和 Reference Model 两份权重，7B 模型显存占用约 30G，单卡根本扛不住。ORPO 把 SFT Loss 和偏好 Loss 写进同一个公式，**不需要 Reference Model**，7B 模型只要 20G 显存就能跑，显存直接省了三分之一。而且 ORPO 一次训练搞定，不用先 SFT 再 DPO 跑两遍，训练时间也省了。对我这种单卡玩家来说，ORPO 就是 DPO 的完美替代方案。" |
| **你用过哪些训练框架？踩过什么坑？** | LLaMA-Factory 快速验证 + Unsloth 单卡加速 + TRL 学原理 | "我平时用三个工具，分场景。**LLaMA-Factory** 用来快速验证想法，点点鼠标就能跑，支持几乎所有主流模型，适合向业务方展示 Demo；**Unsloth** 用来实际训练，它重写了 Triton kernel，训练速度比官方快 2-5 倍，显存还省一半，单卡首选；**TRL** 用来深入理解原理，它的代码最规范，我读 DPOTrainer 的源码来理解 DPO 的梯度推导。坑的话——QLoRA 训练时如果 batch_size 设太大容易爆显存，我就用 gradient_checkpointing 来解决；NEFTune 在某些基座模型上有奇效，但不是所有模型都适用，这个得自己测。" |

---

**最后一句心法**：讲微调方案，**不要只说"我用过 SFT/DPO"，要说"我为什么选它、放弃了什么、踩过什么坑"**。**对比表 + 故事化比喻 + 量化数据 + 追问展开 = 高级感**。

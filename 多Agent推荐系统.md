# 多Agent智能推荐与营销系统

## 项目背景

公司是一家专注于工程机械配件（挖掘机滤芯、液压件、轴承、履带板等）出口贸易的企业，面向东南亚、中东、非洲等海外市场。客户多为海外经销商和代理商，采购决策周期长、品类需求分散、复购率高但客单价也高。传统的"货架式"产品陈列无法精准匹配客户需求，营销文案也依赖人工撰写，效率低且难以本地化。

## 我负责的工作

### 一、需求分析与架构设计

在项目初期，我深入调研了外贸业务团队的痛点：海外客户询价时，业务员需要根据客户的采购历史、所在地区、项目类型快速筛选出最匹配的配件 SKU，同时生成一封专业的英文/阿拉伯语营销邮件。我主导了整体架构设计，将一个复杂的推荐任务拆解为四个独立但协作的 AI 角色：用户画像Agent负责理解"客户是谁"、商品推荐Agent负责回答"推荐什么"、营销文案Agent负责"怎么说"、库存决策Agent负责"能不能卖"。整个调度采用 Supervisor 并行模式，阶段一用户画像和商品初筛并行执行，阶段二精排和库存并行，阶段三生成文案，实现全链路 12 秒内完成。

### 二、用户画像Agent——把客户行为转化为业务洞察

这个 Agent 是整个系统的"大脑"。我设计了两层数据融合机制：离线层用 Redis 存储客户的历史询价记录、采购频次、平均订单金额，实时层记录当前会话中的浏览行为。Agent 基于 RFM 模型（最近一次询价时间、询价频次、累计采购金额）对客户进行价值分层，输出客户是高价值的"战略经销商"还是价格敏感的"新开发客户"，从而指导后续推荐策略的松紧程度。举例来说，一个中东老客户采购液压件多年，系统会自动识别为高价值用户，推荐时优先展示高利润的 OEM 品牌配件而非低价副厂件。

### 三、商品推荐Agent——从海量 SKU 中精准选品

公司有数百个配件 SKU，涵盖发动机件、液压系统、底盘件等数十个类目。推荐Agent 的核心工作分两步：召回和排序。召回阶段，我实现了多策略并行——按客户偏好类目加权、按历史采购关联性排序、按库存深度优先。排序阶段，接入豆包大模型，将客户画像和候选商品打包成结构化 prompt，让模型像一个经验丰富的配件销售经理那样选出最合适的 3 到 5 个 SKU。类目打散逻辑确保推荐结果不扎堆（比如不会连续推三个滤芯），同一卖家也不会霸屏，保证了推荐的专业感和丰富度。

### 四、营销文案Agent——把推荐结果包装成商业价值

外贸场景下，文案的专业度和本地化程度直接影响客户回复率。这个 Agent 根据商品属性和客户画像，自动生成多语言营销内容。例如对价格敏感的非洲客户，文案强调性价比和耐用性；对中东战略客户，强调品牌授权和供应稳定性。所有文案均通过豆包大模型生成，结合 LangChain 的对话记忆能力，保证上下文的连贯性和专业性。营销文案的生成质量通过 A/B Test 引擎持续优化——同一客户群体会收到两套不同风格的文案，系统记录打开率和回复率，Thompson Sampling 算法自动把更多流量分配给表现更好的文案策略。

### 五、库存决策Agent——打通推荐与供应链

推荐出去的产品必须是有货的。这个 Agent 负责实时校验候选商品的库存状态、备货周期和起订量，对库存偏紧的商品主动预警并推荐替代 SKU。某配件缺货时，系统不会直接返回空结果，而是自动切到同类型替代品，确保客户体验的连贯性。

### 六、A/B Test 引擎——数据驱动的策略迭代

我设计并实现了 Thompson Sampling 算法驱动的 A/B Test 引擎，支持多组实验并行。对比测试不同推荐策略（比如"低价优先"vs"品牌优先"）和不同文案风格（技术参数导向 vs 商业价值导向）的效果差异，用后验概率动态调整流量分配，无需人工干预即可持续优化系统效果。

## 项目成果

系统上线后，单次推荐请求全链路响应时间稳定在 12 秒以内，商品推荐命中率明显提升，营销文案平均生成质量获得业务团队认可，整体架构支持 Docker 容器化部署，可横向扩展至日均上万次请求。

## 技术栈

Python / FastAPI / Redis / Milvus 向量检索 / LangChain / LangGraph / 豆包大模型 / Docker-Compose / A/B Testing

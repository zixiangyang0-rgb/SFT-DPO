# 小红书自动化营销系统项目文档

> 适用对象：工程机械外贸从业者 / 全栈开发者
> 项目周期：6 个月以上
> 技术栈：Python + Chrome DevTools Protocol + Claude Code + XiaohongshuSkill

---

## 一、项目背景与问题定义

### 1.1 业务场景

我们公司从事工程机械外贸（主营挖掘机、装载机、压路机等设备），目标市场集中在东南亚、中东和非洲。2024 年下半年，公司决定启动小红书作为海外品牌曝光和潜在客户触达的渠道。

选择小红书的原因有三：第一，小红书用户中 25-40 岁男性比例持续上升，与我们的采购决策者画像高度重合；第二，平台对机械设备类内容的竞争相对较少，流量红利仍在；第三，相比阿里巴巴国际站和 Google Ads，小红书的内容成本低、生命周期长，一条优质笔记可以持续获得搜索流量。

### 1.2 核心痛点

然而，运营初期我们面临严峻的人工瓶颈：

- **发布效率低**：每条笔记从选图、文案撰写、排版到发布，全流程耗时约 40-60 分钟
- **素材管理混乱**：数百张产品图、施工案例视频、客户现场图散落在各个文件夹，没有统一管理
- **账号矩阵难维护**：运营、设计、销售三个角色共用一个账号，操作记录无法追溯
- **页面改版即失效**：小红书前端频繁改版，导致自动化脚本中的 CSS 选择器大量失效，维护成本极高
- **发布频率不足**：人工操作每天最多发布 2-3 条，无法形成持续曝光

我们意识到，要真正用好小红书，必须将发布流程自动化，同时保证账号安全。

## 二、技术方案设计

### 2.1 整体架构

项目采用**本地浏览器直驱**的技术方案，核心技术链路如下：

```
用户指令（Claude Code / CLI）
    ↓
XiaohongshuSkill（任务编排层）
    ↓
Hermes（AI 调度层，负责任务分发与状态管理）
    ↓
cdp_publish.py（CDP 协议层，Chrome DevTools Protocol）
    ↓
Chrome 浏览器（执行层，模拟人工操作）
```

这套架构的核心优势在于：**不依赖小红书官方 API**（官方 API 未对外开放），而是通过 Chrome DevTools Protocol 直接控制浏览器，每次操作等同于真实用户在浏览器中的行为，平台无法从协议层面区分"自动化"和"人工"。

### 2.2 技术选型依据

| 组件 | 选型 | 原因 |
|------|------|------|
| 浏览器控制 | CDP（Chrome DevTools Protocol） | 比 Selenium 更底层，不依赖 WebDriver，不容易被检测 |
| AI 调度 | Claude Code + Hermes | 自然语言指令 → 结构化任务，降低使用门槛 |
| 发布编排 | XiaohongshuSkill | 开源社区专为小红书设计的自动化框架，支持多账号、多任务 |
| 运行环境 | 本地 Python 3.10+ | 可在 Windows / macOS / Linux 运行，无需额外服务器 |
| 账号隔离 | Chrome Profile 独立目录 | 每个角色（运营/设计/销售）使用独立 Profile，Cookie 完全隔离 |

### 2.3 核心模块说明

项目代码主要分布在以下文件中：

- **publish_pipeline.py**：端到端发布管道，接收标题、内容、图片 URL，执行下载-预处理-发布全流程
- **cdp_publish.py**：CDP 协议核心，负责 WebSocket 连接、命令发送、元素定位、异常处理，共 4800+ 行
- **chrome_launcher.py**：Chrome 生命周期管理，支持启动（指定调试端口）、重启、关闭
- **image_downloader.py**：图片下载模块，支持并发下载、重试机制、格式转换
- **account_manager.py**：多账号管理，支持切换 Profile、登录状态持久化、操作日志记录
- **run_lock.py**：单实例锁，防止多进程同时操作同一账号导致冲突

## 三、核心功能实现

### 3.1 多账号矩阵管理

针对运营、设计、销售三个角色各自的需求，我们实现了基于 Chrome Profile 的账号隔离方案。每个角色在独立的 Chrome Profile 下操作，各自的登录状态、Cookie、浏览记录完全独立，通过 `account_manager.py` 统一管理切换。

具体实现上，CDP 连接时通过 `--profile-directory` 参数指定 Profile 路径，账号切换变为"关闭当前 CDP 连接 → 切换 Profile → 建立新连接"的轻量操作，切换时间控制在 3 秒以内。

### 3.2 反检测机制设计

这是整个项目最具技术含量的部分，也是最容易导致账号被封的核心环节。我们实现了三层防护：

**第一层：操作随机化。** 所有 `sleep` 等待不再是固定时长，而是以 `random.uniform(base, base * 1.5)` 的方式抖动。例如基础等待 1 秒，实际等待会在 1.0-1.5 秒之间随机浮动。

**第二层：行为拟人化。** 鼠标移动使用贝塞尔曲线生成轨迹，而非直线位移。点击操作随机偏移 2-3 像素，模拟人手微颤。输入文字不再是瞬间粘贴，而是逐字符输入，间隔 50-150ms 随机。

**第三层：上传操作模拟。** 文件上传是最容易被检测的环节。我们将上传操作拆解为：点击上传按钮 → 等待文件选择对话框出现 → 键入文件路径 → 按回车 → 等待上传进度条消失。单次上传完成后随机停顿 2-5 秒再进行下一次，避免连续快速上传。

这套机制使账号连续稳定运行 6 个月，0 封禁记录。

### 3.3 页面改版容错方案

小红书前端改版频繁，CSS 选择器失效是最常见的问题。我们的应对策略是**多级 fallback + 失效告警**：

```python
# 设计 3 层备选选择器，逐级降级
selectors = [
    "div.publish-left >> textarea.content-input",
    "div[data-v-xxx] >> textarea",
    "//textarea[contains(@placeholder, '分享')]",
]
for selector in selectors:
    element = await find_with_selector(selector)
    if element:
        break
else:
    alert_admin("所有选择器均失效，请人工介入")
```

此外，我们定期运行 `check-login` 命令验证核心选择器可用性，每次改版后优先更新 `SELECTORS` 字典，维护时间控制在 30 分钟以内。

## 五、遇到的挑战与解决思路

### 5.1 图片下载超时

早期版本中，图片下载使用串行请求，50 张图片下载耗时超过 3 分钟，且网络波动时频繁超时。后续引入异步并发下载，利用 `aiohttp` 配合信号量控制并发数，50 张图片下载时间降至 40 秒以内，成功率提升至 99%+。

### 5.2 多进程竞争

当多个脚本同时运行（运营在发布、设计在上传图片），CDP 连接会互相干扰，导致页面状态不一致。通过 `run_lock.py` 实现基于文件锁的单实例运行，任何时刻只允许一个进程操作 Chrome，从根本上消除了竞争条件。

### 5.3 跨平台路径兼容

公司内部同时使用 Windows（设计部）和 macOS（运营部），文件路径格式不同（反斜杠 vs 正斜杠）。在文件上传前统一对路径做规范化处理，Windows 下保留反斜杠以兼容系统调用，macOS/Linux 下转换为正斜杠。

## 六、未来优化方向

以下规划为下一阶段的改进方向，部分已有初步方案：

**方向一：图片并行下载升级为批量上传并发。** 当前发布流程中，图片上传仍是串行执行。计划引入 CDP 的 `Input.dispatchDragEvent` 批量拖拽上传，预期将 9 张图片的上传时间从 90 秒缩短至 20 秒，整体发布流程压缩至 5 分钟以内。

**方向二：素材自动标签化。** 利用 Claude Vision 对产品图片自动识别设备型号、工况场景，生成结构化元数据，解决当前素材库标签混乱的问题，支持按"挖掘机 + 东南亚 + 客户现场"等复合条件筛选。

**方向三：内容 A/B 测试框架。** 同一产品生成多个标题和封面版本，通过自动化脚本分时段发布，根据浏览量、点赞、收藏三维度自动打分，沉淀最优内容模板。

**方向四：数据看板集成。** 对接小红书创作者中心 API（或通过 CDP 抓取个人主页数据），自动汇总每日发布量、浏览量、粉丝变化，生成日报推送至微信/钉钉，解决目前人工统计的效率问题。

## 七、个人成长与反思

这个项目让我在以下几个维度有了显著提升：

**全栈工程能力**：从后端 Python 脚本到前端浏览器自动化，从协议层（WebSocket/CDP）到表现层（CSS 选择器），覆盖了完整的技术栈。在真实业务场景中理解了"为什么需要反检测"——不是技术炫技，而是平台规则倒逼出来的工程约束。

**产品思维**：运营同事提的每一个需求背后都是一个真实痛点。比如"能不能让我只上传图片，让 AI 自动生成标题和内容"——这直接推动了内容生成模块的规划。技术方案必须服务于真实的业务目标，而不是追求技术的复杂度。

**跨角色协作**：作为全栈开发者，我需要理解运营的内容策略、设计的素材规范、销售的询盘转化逻辑。只有深入理解业务上下文，才能写出真正有用的自动化脚本。

## 附录：快速上手指南

如需复现或二次开发，以下是关键依赖和环境配置：

```bash
# 1. 克隆项目
git clone <repo-url>
cd XiaohongshuSkills-main

# 2. 创建虚拟环境
python -m venv .venv && source .venv/bin/activate

# 3. 安装依赖
pip install -r requirements.txt

# 4. 配置账号
cp config/accounts.json.example config/accounts.json
# 编辑 accounts.json，填入 Chrome Profile 路径

# 5. 启动 Chrome
python scripts/chrome_launcher.py

# 6. 验证登录状态
python scripts/cdp_publish.py check-login

# 7. 执行发布
python scripts/publish_pipeline.py --title "xxx" --content "xxx" --image-urls "https://xxx.jpg"
```

详细 API 文档和命令行参数说明请参考项目根目录的 `SKILL.md` 和 `README.md`。

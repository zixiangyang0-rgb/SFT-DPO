# 项目经历：AI 多 Agent 跨境运营平台

**公司背景**：某工程机械外贸出口企业，主营挖掘机、装载机、压路机及零配件出口，业务覆盖东南亚、中东、非洲、拉美等 30+ 国家与地区。海外订单分散在多套区域 ERP，海外仓库存与海运/陆运物流跨多个时区难统一跟踪，采购商咨询、售后响应链路长，传统 BI 报表查询门槛高、滞后严重。

**岗位**：AI 应用开发

**时间**：2025.XX – 2026.XX

---

## 项目背景与痛点

公司月均出口订单 800+，SKU 1200+，采购商客户 400+，日常运营高度依赖销售、跟单、客服三条业务线线下协作。具体痛点包括：

1. **数据散落**：销售数据在区域 ERP、库存数据在 WMS、物流数据在货代系统，运营查一个"上周中东区订单 + 在途库存"需要在 3 个后台切换；
2. **响应滞后**：海外采购商通过邮件、WhatsApp 咨询，客服需要人工翻单、查库存，跨时区沟通下响应普遍超过 24 小时；
3. **预警缺位**：海外仓断货、滞留、退货异常几乎都是事后发现，错过补货窗口期，单次缺货损失 5–15 万元；
4. **报表门槛高**：业务侧 BI 工具学习成本高，运营人员遇到定制化需求需排期开发，需求从提出到上线平均 2 周。

## 方案设计

基于 LLM + Skill（技能）机制搭建 AI 运营中台，将传统 BI 报表重构为"对话式运营入口"，并把能力构建从"编译型"改为"解释型"，运营人员可直接上线新能力，无需工程师介入。

### 1. 业务能力抽象为 Skill

自研 4 个核心 Skill，用 Markdown 文件描述数据查询逻辑，底层通过 `curl + jq` 封装业务 API：

- **海外订单 Skill**：按区域、机型、客户、币种维度查询订单概况与明细，支持同比/环比对比；
- **海外仓库存 Skill**：覆盖东南亚、中东、非洲三大主仓的低库存预警、断货提醒、补货建议；
- **采购商洞察 Skill**：新客/回头客分析、地域分布、采购周期画像；
- **机型热销 Skill**：按机型、功率段、出口目的国的热销榜与滞销品分析。

所有 Skill 放在对应 Agent 的 `skills/` 目录下，分钟级上线，无需发版。

### 2. 多 Agent 角色分工

搭建 3 个 Agent，分别绑定飞书部门群，通过 SOUL.md 的人格定义实现角色隔离：

| 飞书群        | 绑定 Agent | 加载 Skills                | 人格                |
|-------------|-----------|---------------------------|-------------------|
| 海外销售部      | sales     | 海外订单、机型热销             | 数据优先、结论先行         |
| 跟单部         | tracker   | 海外仓库存、物流跟踪            | 时效优先、节点清晰         |
| 海外客服部      | support   | 采购商洞察、工单查询             | 耐心专业、安抚为主         |

不同 Agent 拥有独立 Workspace（AGENTS.md / SOUL.md / MEMORY.md / skills/），互不干扰。每个用户按 `senderOpenId` 拥有独立 Session，对话历史互相隔离。

### 3. Cron 主动通知取代被动查询

按海外时区与业务节奏配置定时任务，把 AI 从"你问我答"升级为"主动告诉你"：

- **每日 8:00（北京时间）**：聚合昨日各区域订单概况、异常订单、低库存预警，生成早报推送到销售部群；
- **每 30 分钟**：扫描海外仓库存，断货或低于安全库存即推送到跟单部群；
- **每周一 9:00**：输出 TOP 10 热销机型与滞销品周报，辅助备货决策；
- **实时**：大额退款、客诉升级即时告警。

### 4. 安全与成本控制

- **写操作严格隔离**：所有 Skill 默认只读，退款、改价等操作在业务 API 侧加人工审批，AI 层只做信息汇总；
- **凭证管理**：API Token 通过环境变量注入，不硬编码在脚本或配置文件中；
- **成本分级**：简单查询使用 Haiku 模型（约 Sonnet 成本的 1/10），复杂分析才升级到 Sonnet；
- **Prompt Cache**：对高频前缀启用缓存，节省 80–90% 的 input token 费用。

## 关键产出与技术亮点

- **独立负责** 4 个 Skill 的设计、开发与文档化，覆盖订单、库存、客户、商品 4 大场景；
- **搭建** 多 Agent 路由配置（`bindings` + `dmScope: per-peer`），实现"一个 AI 引擎，多个角色入口"；
- **落地** 飞书 WebSocket 渠道接入，无需公网入站端口，部署在内网 K8s 集群；
- **探索** SaaS 多租户架构（Per-tenant Pod + Envoy Gateway），为后续平台化对外赋能预留扩展空间；
- **工程化** Skill 模板与 SOUL.md 模板沉淀，使非技术人员也能分钟级封装新能力。

## 效果与收益

| 指标               | 上线前           | 上线后      | 变化          |
|------------------|---------------|----------|-------------|
| 运营日均查数次数         | 40+（后台切换）    | 10+（群内对话）| -75% 切换成本   |
| 缺货预警发现时间         | 平均 3.2 天      | < 2 小时   | -97%         |
| 采购商咨询平均首响         | 24h+          | < 30 分钟  | -95%         |
| 简单查询单次成本         | —             | ~$0.08    | —            |
| 复杂分析单次成本         | —             | ~$0.12    | —            |
| 基础设施分摊（100 商户规模） | —             | ~$17.7/月  | —            |

业务侧反馈最直观的 3 个提升：

1. **运营查数不再切后台**——群里一句话拿到结论，BI 工具几乎不再被使用；
2. **海外客服响应提速 95%**——采购商咨询通过飞书直连 AI，复杂工单再转人工；
3. **备货决策有数据支撑**——周报成为业务例会固定输入，滞销品主动清仓，库存周转率提升约 18%。

## 总结与反思

项目验证了"LLM + Skill + 多 Agent 飞书路由"在传统外贸企业落地的可行性。最大启示是：**能力构建从编译型变成解释型**——运营人员会用 `curl + jq` 就能封装一个 Skill，业务迭代速度从"两周排期"压缩到"分钟级上线"。

后续可继续推进的方向包括：引入竞品价格监控（对接海外 B2B 平台）、基于历史订单的备货预测模型、跨境多语种自动翻译（英/俄/阿/西）等。平台层会逐步从"内部工具"演进为"对外赋能的 SaaS"，支撑更多中小外贸企业以更低成本接入 AI 运营能力。

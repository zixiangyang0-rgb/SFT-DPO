### 3\.2 Step 1：微信数据库解析导出（01\_export\_wx\.py）

```Python
import sqlite3
import json
from datetime import datetime

def parse_wx_db(db_path: str) -> list[dict]:
    """解析微信消息数据库，提取纯文本消息"""
    conn = sqlite3.connect(db_path)
    cursor = conn.cursor()

    cursor.execute("""
        SELECT localId, TalkerId, IsSender, Type, StrContent, 
               CreateTime, StrTalker
        FROM MSG 
        WHERE Type = 1           -- 仅文本消息
          AND StrContent != ''   -- 非空
        ORDER BY CreateTime ASC
    """)

    messages = []
    for row in cursor.fetchall():
        messages.append({
            "msg_id": row[0],
            "talker_id": row[1],
            "is_sender": row[2],       *# 1=顾问, 0=学员*"content": row[4],
            "timestamp": row[5],
            "talker_wxid": row[6],
            "time_str": datetime.fromtimestamp(row[5]).strftime("%Y-%m-%d %H:%M:%S")
        })

    conn.close()
    return messages
```

### 3\.3 Step 2：会话切分（02\_session\_split\.py）

按**对话对象 \+ 时间间隔**切分独立会话：

```Python
SESSION_GAP_MINUTES = 120  *# 超过2小时视为新会话*def split_sessions(messages: list[dict]) -> list[list[dict]]:
    """将消息流按对话对象和时间间隔切分为独立会话"""
    grouped = defaultdict(list)
    for msg in messages:
        grouped[msg["talker_wxid"]].append(msg)
    
    all_sessions = []
    for wxid, msgs in grouped.items():
        msgs.sort(key=lambda x: x["timestamp"])
        current_session = [msgs[0]]
        
        for i in range(1, len(msgs)):
            gap = (msgs[i]["timestamp"] - msgs[i-1]["timestamp"]) / 60if gap > SESSION_GAP_MINUTES:
                all_sessions.append(current_session)
                current_session = []
            current_session.append(msgs[i])
        
        if current_session:
            all_sessions.append(current_session)
    
    return all_sessions
```

- 原始 5,200 段对话 → 切分为 **8,347 个独立会话**

- 平均每个会话 **10\.3 条消息**

### 3\.4 Step 3：清洗过滤（03\_clean\_filter\.py）

这是最关键的步骤，做了 **7 层过滤**：

```Python
class DataCleaner:
    def clean(self, sessions: list) -> list:
        results = []
        for session in sessions:
```

#### ③ 合并连续同一方消息

微信中经常出现一个人连发多条短消息的情况，合并为一条：

```Python
def merge_consecutive(self, session):
    """将连续同一方的消息合并，间隔<60s的拼接"""
    merged = []
    for msg in session:
        if (merged and 
            merged[-1]["is_sender"] == msg["is_sender"] and
            msg["timestamp"] - merged[-1]["timestamp"] < 60):
            merged[-1]["content"] += "\n" + msg["content"]
        else:
            merged.append(msg)
    return merged
```

### 4\.2 从多轮对话构建 SFT 样本

每段多轮对话会被展开成**多个训练样本**（滑动窗口方式）：

```Python
def session_to_sft_samples(session: list[dict], system_prompt: str) -> list[dict]:
    """一段对话生成多个SFT样本，每个顾问回复作为一个output"""
    samples = []
    
    for i, msg in enumerate(session):
        if msg["is_sender"] == 1:  *# 顾问回复*
            history = session[:i]
            if not history:
                continue
            
            history_text = ""for h in history[-10:]:  *# 最多保留最近10轮上文*
                role = "学员" if h["is_sender"] == 0 else "顾问"
                history_text += f"{role}: {h['content']}\n"
            
            samples.append({
                "instruction": system_prompt,
                "input": history_text.strip(),
                "output": msg["content"]
            })
    
    return samples
```

### 5\.2 DPO 数据构建方式

采用 **三种方式混合** 生成 DPO 对比对：

#### 方式一：同一问题的不同顾问回复（占 40%）

同一类问题，不同顾问的回答质量参差不齐。把质量高的作为 chosen，质量低的作为 rejected：

```Python
def build_dpo_from_multi_advisor(similar_qa_pairs):
    """同类问题的不同顾问回复，优质回复为chosen，一般回复为rejected"""
    dpo_samples = []
    for group in similar_qa_pairs:
        scored = [(qa, score_by_gpt4(qa)) for qa in group]
        scored.sort(key=lambda x: x[1], reverse=True)
        
        if len(scored) >= 2 and scored[0][1] - scored[-1][1] >= 1.5:
            dpo_samples.append({
                "instruction": SYSTEM_PROMPT,
                "input": scored[0][0]["input"],
                "chosen": scored[0][0]["output"],    *# 高分回复*"rejected": scored[-1][0]["output"],  *# 低分回复*
            })
    return dpo_samples
```

#### 方式二：GPT\-4 生成改写（占 35%）

对 SFT 数据中质量一般的 output，用 GPT\-4 生成一个更好的版本：

```Python
REWRITE_PROMPT = """你是一位资深课程销售培训师。
```

#### 方式三：人工标注（占 25%）

### 7\.4 数据转换脚本

```Python
def alpaca_to_ark_sft(alpaca_data: list[dict]) -> list[dict]:
    """将本地 Alpaca 格式转为方舟平台 messages 格式"""
    ark_data = []
    for item in alpaca_data:
        messages = [
            {"role": "system", "content": item["instruction"]},
        ]
        
        turns = item["input"].split("\n")
        for turn in turns:
            if turn.startswith("学员:") or turn.startswith("学员: "):
                messages.append({"role": "user", "content": turn.split(": ", 1)[1]})
            elif turn.startswith("顾问:") or turn.startswith("顾问: "):
                messages.append({"role": "assistant", "content": turn.split(": ", 1)[1]})
            else:
                messages.append({"role": "user", "content": turn})
        
        messages.append({"role": "assistant", "content": item["output"]})
        ark_data.append({"messages": messages})
    
    return ark_data
```

## 九、关键经验总结（面试高频追问）

### Q1：微调和 Prompt Engineering / RAG 有什么区别？什么时候该用微调？

```Plain Text
Prompt Engineering：改变模型"怎么理解问题"，不改变模型本身
RAG：给模型"喂外部知识"，解决知识更新问题
微调：改变模型"怎么说话"，从根本上调整模型行为

我们三者都用了：
- System Prompt：定义 AI 人设的基本规则
- RAG：注入课程价格、课纲等实时信息
- 微调：让模型学会课程顾问的对话风格和转化技巧

关键判断标准：如果通过 Prompt 已经能达到 80% 的效果，就不需要微调。
我们微调主要解决的是 Prompt 解决不了的问题：话术转化能力、异议处理的自然度。
```

### Q2：数据质量如何保证？

```Plain Text
三道防线：
1. 规则过滤：去系统消息、去噪声、去短对话、脱敏
2. 模型打分：GPT-4o-mini 批量质量评分，≥3 分才保留
3. 人工抽检：每轮训练前随机抽 200 条人工复核，标注准确率要求 >90%
```

### Q3：怎么判断微调效果好不好？

```Plain Text
三个层面：
1. 训练指标：train loss 下降、eval loss 不回升（不过拟合）、DPO reward margin 上升
2. 人工评测：100 条标准测试集，5 个维度打分，盲评对比
3. 线上指标：A/B 测试看对话轮次、留资率、满意度
```

### Q4：为什么先本地再上云？

```Plain Text
成本和效率考虑：
- 本地跑一轮 2-3 小时，免费，可以快速迭代
- 云端一轮 4-6 小时，有计算费用，而且要排队
- 数据格式有问题、数据质量不行这些低级错误，在本地就能发现
- 本地用 Qwen 7B 验证数据方向，确认可行后再上云端用 Doubao 1.5 正式训练

相当于本地是"开发环境"，云端是"生产环境"
```

### Q5：RTX 5060 只有 16GB，怎么跑 7B 模型？

```Plain Text
三个技巧：
1. QLoRA 4-bit 量化：模型权重从 FP16 压缩到 4-bit，显存占用降 75%
2. Gradient Checkpointing：用时间换空间，训练显存降 40%
3. FlashAttention 2：RTX 5060 支持 FA2，注意力计算显存降 50%

三个技巧叠加后，7B 模型 LoRA 微调只需要 ~12-15GB 显存，16GB 刚好够用。
```

### Q6：DPO 的 β 参数怎么选的？

```Plain Text
β 控制偏好学习的强度：
- β 大（如 0.5）：模型变化小，保守
- β 小（如 0.01）：模型变化大，激进

我们选 0.1 是因为：
- 我们的 SFT 基础已经不错了，不需要 DPO 做太大改变
- 0.1 是业界常用的默认值
- 试过 0.05（reward margin 上升太快，有过优化风险）和 0.2（效果不明显）
```

# 主流模型微调方案解读

# 业内的几种模型微调方案

SFT \(Supervised Fine\-Tuning\) \- 有监督微调

DPO \(Direct Preference Optimization\) \- 直接偏好优化

PPO \(Proximal Policy Optimization\) \- 近端策略优化 **极其难训**！显存占用巨大

OPO \(通常指 ORPO\)Odds Ratio Preference Optimization\) \- 赔率比偏好优化 SFT 和 DPO 二合一



# 有监督微调 \(SFT\)

大模型进化的三个阶段：Pre\-train vs SFT vs RLHF

#### 第一阶段：预训练 \(Pre\-training \) 

- **本质**：海量数据投喂，学习“接龙”。

- **数据**：万亿级 Token 的通用文本（维基百科、GitHub 代码、网页、书刊）。

- **目标**：让模型学会语言的语法、逻辑和世界知识。

- **状态**：此时的模型（Base Model）虽然博学，但**不懂指令**。

    - *你问它*：“写首诗。”

    - *它可能续写*：“\.\.\.写首歌，写篇散文。”（它以为你在跟它玩成语接龙）

- **门槛**：极高。需要数千张 GPU 训练数月（如 Llama 3 Base, Qwen Base）。**个人开发者通常跳过此阶段，直接使用开源基座。**

#### 第二阶段：有监督微调 \( SFT\) 

- **本质**：教模型“听懂人话”，学会按指令格式输出。

- **数据**：高质量的 `(Instruction, Input, Output)` 问答对。

- **目标**：激活模型在预训练阶段学到的知识，使其适配特定任务（如医疗问诊、法律咨询、写代码）。

- **状态**：此时的模型（Chat Model）可以流畅对话，但可能**容易胡说八道（幻觉）**，或者**不懂礼貌**（输出有害信息）。

- **门槛**：中低。单卡或多卡即可完成。

#### 第三阶段：人类价值观对齐 \(RLHF / Alignment\) DPO

- **本质**：通过奖惩机制，规范模型的行为边界。

- **数据**：人类偏好数据 `(Chosen, Rejected)`，告诉模型哪个回答更好。

- **目标**：让模型**有用（Helpful）、诚实（Honest）、无害（Harmless）**。减少幻觉，拒绝回答违法问题。

- **状态**：这是 ChatGPT 区别于普通聊天机器人的关键。

- **门槛**：中高。需要精细的数据和复杂的算法（PPO/DPO）。



### 举个例子：SFT和DPO的区别

### 第一幕：SFT（有监督微调）—— 像“教条”一样教做菜

**解决的问题**：教模型\*\*“懂规矩”\*\*。让它知道当你问“怎么做红烧肉”时，它应该给出一份菜谱，而不是开始写一首关于猪的诗。

**故事场景**： 你（导师）坐在徒弟旁边，拿出一本《满汉全席标准教程》。你告诉他：“看好了，如果客人说‘**我想吃辣的**’，你就得给‘**麻婆豆腐**’，步骤是1、2、3。”

**SFT 案例**：

- **输入（Instruction）**：请帮我写一段电商评论的情绪分析代码。

- **输出（Standard Answer）**：`import pandas as pd ...`（一段规范、完整的 Python 代码）。

- **效果**：徒弟学会了**格式**。问他问题，他不再乱说话，而是像个专业助手一样回答。

### 第二幕：DPO（偏好对齐）—— 培养“厨德”与“情商”

**解决的问题**：解决\*\*“好坏”**和**“边界”\*\*问题。SFT 只教了“怎么做”，但没教“怎么做更好”或者“什么不能做”。

**故事场景**： 徒弟现在会做菜了，但他有时会往菜里加砒霜（有害信息），或者虽然菜熟了但特别难吃（虽然对但废话多）。 这时候你给他两个盘子：

- **盘子 A**：色香味俱全。

- **盘子 B**：太咸了，或者摆盘很乱。 你不需要告诉他怎么做，你只需指着盘子说：**“选 A，别选 B。”**

**DPO 案例（对比学习）**：

- **输入**：我买的衣服质量太差了，气死我了！

- **回答 A \(Chosen/优选\)**：我很抱歉听到这个消息。您可以联系客服申请退货，或者我可以帮您写一段投诉建议？（**有用且有礼貌**）

- **回答 B \(Rejected/拒绝\)**：衣服差就差呗，你自己选的。（**虽然是中文，但有毒、没礼貌**）

- **效果**：通过大量 A \> B 的选择，模型学会了：**哦！原来人类喜欢温和、专业的回答，讨厌刻薄和复读。**

# SFT 数据处理

在 AI 圈有一句至理名言：“**Garbage In, Garbage Out**”（垃圾进，垃圾出）。

模型微调的效果，**70% 取决于数据质量，30% 取决于算法和参数**。

很多时候，你调了一个月的模型参数，效果还不如花一天时间清洗一下数据来得明显。

### 两大主流数据格式：Alpaca vs ShareGPT

在收集数据之前，你必须先决定把数据存成什么样子。目前微调界主要有两种标准格式。

格式参考官方文档：https://llamafactory\.readthedocs\.io/zh\-cn/latest/getting\_started/data\_preparation\.html

#### Alpaca 格式 \(单轮指令微调的标准\)

源自斯坦福的 Alpaca 项目。最适合\*\*“一问一答”\*\*的任务，如逻辑推理、文本分类、翻译。

- **结构**：

    - `instruction`: 任务描述（必填）。

    - `input`: 补充信息（选填，比如需要翻译的原文）。

    - `output`: 模型的理想回答（必填）。

- **JSON 示例**：

```JSON
[
  {
    "instruction": "请将下面的句子翻译成英文。",
    "input": "今天天气真好。",
    "output": "The weather is nice today."
  },
  {
    "instruction": "解释量子纠缠的概念。",
    "input": "",
    "output": "量子纠缠是指..."
  }
]
```

#### ShareGPT 格式 \(多轮对话微调的标准\)

源自 ShareGPT 项目（收集用户与 ChatGPT 的对话）。最适合“聊天机器人”，能让模型学会联系上下文。

目前主流框架（如 LLaMA\-Factory）都首选支持这种格式。

- **结构**：

    - `conversations`: 一个列表，包含多轮对话。

    - `from`: 角色（通常是 `human`/`user` 和 `gpt`/`assistant`）。

    - `value`: 对话内容。

- **JSON 示例**：

```JSON
[
  {
    "conversations": [
      {
        "from": "human",
        "value": "你好，我想买一台电脑，有什么推荐？"
      },
      {
        "from": "gpt",
        "value": "您好！请问您主要用来做什么？玩游戏还是办公？预算多少？"
      },
      {
        "from": "human",
        "value": "主要是办公，预算5000左右。"
      },
      {
        "from": "gpt",
        "value": "那推荐您考虑轻薄本，例如..."
      }
    ]
  }
]
```

# 数据清洗：

网上爬下来的数据或者开源数据集，通常充满了“杂质”。直接喂给模型，模型会“中毒”。

#### 去重 

- **为什么要做**：如果模型在训练集中看了 100 次同样的笑话，它就会死记硬背这个笑话（过拟合），导致泛化能力下降。

- **怎么做**：

    - **精确去重**：简单的字符串匹配，删掉完全一样的。

    - **模糊去重 \(MinHash \+ LSH\)**：这是大厂标配。比如“今天天气不错”和“今天天气真不错”，虽然字不一样，但 MinHash 能算出它们极其相似，从而删掉其中一条。

#### 去除“拒绝回答”

很多开源数据是用 GPT\-4 生成的，GPT\-4 经常会说：“作为 AI 语言模型，我无法回答\.\.\.”。

**千万不能把这些数据喂给你的模型！** 否则你的模型也会变成一个只会道歉的复读机。

- **清洗策略**：编写脚本扫描 `output` 字段，如果包含 "As an AI language model" 或 "我无法回答"，直接整条数据丢弃。

#### 关键词过滤

过滤掉含有暴力、色情、广告代码的数据。

这是个玄学问题，但有一些经验法则：

### 数量 \< 质量

- **LIMA 论文证明**：仅用 **1000 条** 顶级质量的人类精修数据，微调出来的模型效果，可以吊打用 **50000 条** 粗制滥造数据微调的模型。

- **策略**：与其疯狂爬数据，不如花钱请专业标注员（或你自己）精修 500 条完美的问答对。

### 配比策略 

不要只放垂直领域数据！

如果你只喂医疗数据，模型可能会“变傻”，忘了怎么写代码，怎么闲聊（灾难性遗忘）。

- **黄金配比**：

    - **70\-80%**：垂直领域数据（如医疗问诊）。

    - **10\-20%**：通用能力数据（Alpaca GPT\-4 通用数据集，维持智商）。

    - **5\-10%**：自我认知数据（告诉它“我是XX医疗助手”，防止它说“我是 ChatGPT”）。

### 数据来源推荐

- **通用能力**：`alpaca_gpt4_zh` \(中文\), `ShareGPT_Vicuna_unfiltered`。

- **自我认知**：需要自己写脚本生成几百条，如 `{"input": "你是谁？", "output": "我是极客医疗助手..."}`。



# SFT 如何更高效？Lora

#### 为什么不推荐全量微调 \(Full Fine\-Tuning\)？

全量微调意味着你要更新模型里的**每一个参数**。

- **成本**：极高。你需要加载模型权重 \+ 梯度 \+ 优化器状态，显存需求是模型大小的 4 倍以上。

- **灾难性遗忘**：为了学会新知识，模型可能会“洗脑”过度，把旧知识忘光了。

#### LoRA 的“旁路”哲学

LoRA 的思想非常天才。它**冻结**住原来的大模型（不许动），只在旁边**外挂**两个非常小的矩阵（A 和 B）来学习新知识。

- **形象比喻**：

- 大模型就像一本**百科全书**。全量微调是把书里的字擦掉重写（费劲、容易写坏）。

- LoRA 是在书页的边角**贴便利贴**（轻松、随时可撕）。推理的时候，我们既看书里的字，也看便利贴上的内容。

- **数学原理（极简版）**：

- 假设原来的权重是 `W`，我们不直接改 `W`，而是训练一个增量 `Delta W`。

- LoRA 假设这个 `Delta W` 可以分解为两个低秩矩阵的乘积：`Delta W = A \times B`。

    - `A` 和 `B` 非常小，参数量可能只有原模型的 1% 甚至 0\.1%。

    - **显存节省**：我们只需要训练 A 和 B，梯度和优化器状态极小。

### QLoRA：把大象装进冰箱

如果说 LoRA 让你能训练模型，那 QLoRA \(Quantized LoRA\) 就是让你能**在烂显卡上**训练模型。

#### 4\-bit 量化 \(Quantization\)

- **原理**：大模型通常用 16\-bit \(FP16/BF16\) 存储。QLoRA 把模型强行压缩到 4\-bit \(NF4 格式\)。

- **效果**：模型体积直接缩小 4 倍！

    - 7B 模型原本占 14G 显存，4\-bit 量化后只占 3\.5G 左右。

- **代价**：精度会有极其微小的损失，但在微调任务中几乎可以忽略不计。

**🚀 总结**：**QLoRA = 冻结的 4\-bit 基座模型 \+ LoRA 适配器**。这是目前单卡训练的标准答案。

### 调参：核心参数详解（重要）

在使用 LLaMA\-Factory 或 Unsloth 训练时，你会面对一堆参数。别慌，真正决定生死的只有这几个。

#### Rank \(r\) 

100w参数模型，1000\*1000

- **含义**：LoRA 矩阵的秩。决定了微调模型能学到多少新知识的“容量”。

- **常见值**：8, 16, 32, 64, 128。

- **经验法则**：

    - **简单任务**（如改说话风格、分类）：`r = 8` 或 `16` 足够。

    - **复杂任务**（如学习全新的医疗知识、逻辑推理）：`r = 64` 或 `128`。

    - **注意**：r 越大，显存占用越高，训练越慢，且容易过拟合。一般 **64** 是个性价比极高的选择。

#### Alpha \(lora\_alpha\) 

- **含义**：缩放因子。决定了新学的知识（LoRA）对原模型的影响有多大。

- **黄金法则**：**Alpha = 2 \* Rank**。

    - 如果 `r = 64`，那么 `alpha` 设为 `128`。

    - 这是无数实验总结出来的“无脑设置”，照做即可。

#### Target Modules 

- **含义**：你要在哪些层挂载 LoRA？

- **Transformer 结构**：主要包含 `q_proj`, `k_proj`, `v_proj`, `o_proj` \(Attention 层\) 和 `gate_proj`, `up_proj`, `down_proj` \(MLP 层\)。

- **最佳实践**：**All Linear Modules \(所有线性层\)**。

    - 以前大家为了省显存只训练 `q_proj` 和 `v_proj`。

    - 现在的研究证明，**把所有层都打开（Train all linear layers）**，效果提升非常明显，而显存增加并不多。

#### Learning Rate \(学习率\)

- **QLoRA 推荐值**：`2e-4` \(0\.0002\)。

    - 这比全量微调（通常 1e\-5）要大，因为 LoRA 参数少，需要更激进的更新。

- **Scheduler**：推荐 `cosine` \(余弦退火\) 或 `linear`。

想象你在开车下坡，**Learning Rate \(LR\)** 是车速，而 **Scheduler** 就是你踩刹车的方式。

### 进阶技巧：NEFTune \(给 Embeddings 加点噪\)

- **全称**：Noisy Embeddings Fine Tuning。

- **原理**：在训练过程中，给输入的 Embedding 向量随机加上一点点噪声。

- **效果**：

    - 防止模型死记硬背（过拟合）。

    - **显著提升对话的流畅度和指令遵循能力**（亲测有效）。

- **用法**：在 Trainer 中设置 `neftune_noise_alpha = 5`。



# 偏好对齐（为给DTO数据）

SFT解决了“让模型说话”的问题，但它无法解决“让模型说好话”的问题。

SFT 后的模型可能会出现以下“坏毛病”：

- **啰嗦复读**：车轱辘话来回说。

- **胡说八道**：为了补全句子而编造事实（幻觉）。

- **缺乏安全感**：用户问如何制造炸弹，它真的去教（缺乏安全性）。

要治好这些毛病，我们需要进入 **Alignment（对齐）** 阶段。而对齐的核心，不是写更多的标准答案，而是告诉模型：**“这个回答比那个更好”**。

本章将教你如何构建 DPO/RLHF 的燃料——**偏好数据集 \(Preference Dataset\)**。

### 核心结构：三元组

与 SFT 的“一问一答”不同，偏好数据通常包含三个部分，我们称之为 **“三元组”**：

1. **Prompt \(提示词\)**：用户的提问。

2. **Chosen \(胜者\)**：人类（或强模型）认为更好的回答。

3. **Rejected \(败者\)**：人类认为较差的回答。

#### 为什么需要 Rejected？

模型就像一个小孩子。

- **SFT** 是你教它：“见到人要说你好”。

- **DPO/RLHF** 是你教它：“见到长辈叫叔叔（Chosen），叫老头子（Rejected）是不礼貌的”。

- **只有同时看到了正例和负例，模型才能通过对比，学到隐含的“价值观”和“风格”。**

#### 标准数据格式 \(JSONL\)

这是目前 Hugging Face `TRL` 库和 `Unsloth` 训练 DPO 时通用的标准格式：

```JSON
{
  "prompt": "如何像海盗一样打招呼？",
  "chosen": "阿霍伊！老伙计！最近在那片大海上发财啊？🏴‍☠️",
  "rejected": "你好，我的朋友。请问最近过得怎么样？"
}
```

- **解析**：两个回答在语义上都对，但 `Chosen` 更好地遵循了“像海盗一样”的风格指令，而 `Rejected` 虽然礼貌但跑题了（风格不符）。

### 避坑指南：数据里的“剧毒”

DPO 算法对数据非常敏感。如果数据里有“毒”，模型会学坏。以下是三个最致命的坑。

#### 长度偏见

- **现象**：模型天生认为\*\*“写的长 = 写的好”\*\*。

- **事故**：如果你构造的数据集中，`Chosen` 总是比 `Rejected` 长，DPO 训练后的模型会变成一个**话痨**。它会为了讨好奖励机制，疯狂堆砌废话。

- **解药**：

    - 在构造数据时，**刻意保留一些“短而精”的 Chosen**。

    - 如果 GPT\-4 选了长的那个，检查一下是不是因为长才选的。如果是废话文学，人工修正为 Rejected。

#### 事实错误 vs 风格差异

- **误区**：用 DPO 来修事实错误（Factuality）。

    - *例子*：Prompt: "1\+1=?" \| Chosen: "2" \| Rejected: "3"

- **真相**：**DPO 更擅长修风格（Style）和安全性（Safety），而不是知识。**

    - 知识最好在 Pre\-train 或 SFT 阶段注入。

    - DPO 阶段如果用来修知识，效率很低，而且容易导致模型“死记硬背”。

#### 相似度过高

- **现象**：Chosen 和 Rejected 只有一两个字的差别。

- **后果**：DPO 算法算出来的 Loss 会非常小，模型学不到东西，甚至梯度消失。

- **解药**：确保两个回答在**逻辑、语气或结构**上有明显的差异（Edit Distance 足够大）。

# DPO 的局限与展望

SFT\+DPO是现在的主流方案。



DPO 虽好，但它不是万能的。在实际的工业界落地中，我们经常遇到三个令人头大的问题：

1. **数据难搞**：构建成对的 `(Chosen, Rejected)` 数据太贵、太难了。能不能只有“点赞”数据？

2. **过拟合**：DPO 有时用力过猛，模型为了讨好人类偏好，导致语言能力退化（开始胡言乱语）。

3. **流程繁琐**：先做 SFT，再做 DPO，要训两次，还要加载 Reference Model，显存和时间双倍消耗。

为了解决这些问题，** ORPO**。

### ORPO \(Odds Ratio PO\)：SFT \+ DPO 二合一

#### 痛点：两阶段训练太慢

传统流程：`Base -> SFT -> Chat Model -> DPO -> Aligned Model`。

这意味着你要跑两遍训练流程，准备两份数据。

#### ORPO 的原理

ORPO 发现，其实我们可以在做 SFT（学习怎么说话）的同时，顺便把 DPO（学习怎么做人）给做了。

它修改了 SFT 的 Loss 函数：

- **Loss = SFT Loss \+ λ \* OR Loss \(Odds Ratio\)**

- **SFT Loss**：保证模型能学会 Chosen 回答（能说话）。

- **OR Loss**：强迫模型生成 Chosen 的概率比 Rejected 高（有偏好）。

#### 优势

1. **省钱**：只需要训练一次。

2. **省显存**：**不需要加载 Reference Model！** 这对显存捉襟见肘的单卡玩家是巨大的福音。7B 模型 DPO 可能要 30G 显存，但 ORPO 只要 20G。
